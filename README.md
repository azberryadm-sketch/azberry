# نظام تقييم الأفرع - ازبيري

نظام ويب متكامل لإدارة زيارات وتقييم الأفرع، مبني بـ Next.js + قاعدة بيانات SQLite (ملف واحد، بدون سيرفر قاعدة بيانات منفصل).

## المميزات المنفذة
- تسجيل دخول بيوزر وباسورد لكل موظف، مع صلاحيتين: **مدير النظام** (كل الصلاحيات) و**موظف تقييم** (يرى فقط الأفرع المخصصة له إن وُجدت).
- إدارة الأفرع (إضافة/تعديل/إيقاف/حذف) — قابل للتوسّع لأي عدد أفرع.
- إدارة المستخدمين وربط كل موظف بالأفرع التي يزورها.
- إدارة **أقسام التقييم** ديناميكياً من داخل النظام (تقدر تضيف أو توقف أي بند تقييم بدون تعديل كود) — معبأة مسبقاً بالبنود التي ذكرتها (الواجهة، المكائن، الصالة، المطبخ، غرف التبريد والتجميد، المخزن...).
- نموذج تقييم بالدرجات (1-5) لكل قسم + ملاحظة نصية اختيارية لكل بند + ملاحظات عامة.
- حساب تلقائي للنسبة المئوية الإجمالية لكل زيارة.
- **داشبورد**: عدد الأفرع، إجمالي الزيارات، أفضل فرع وأسوأ فرع (بالمتوسط)، حالة كل الأفرع بالألوان (أخضر/أصفر/برتقالي/أحمر)، وأضعف الأقسام على مستوى كل الأفرع.
- **تنبيهات تلقائية**: أي قسم تقييم ينزل عن الحد الأدنى المحدد له (قابل للتعديل لكل قسم) ينشئ تنبيه تلقائي، وكذلك أي تقييم عام أقل من 60%.
- **قسم المشاكل**: تسجيل مشاكل (من مشرف الفرع / إدارة الفرع / ملاحظات على الموظفين والشيفات) مع تتبع الحالة (محلولة / غير محلولة).
- تصدير كل تقييم كملف **PDF**.
- تصميم متجاوب (Responsive) يعمل على الحاسبة والموبايل.
- ألوان الواجهة مبنية على هوية الشركة (برتقالي/أبيض) — تقدر تُغيّرها من `lib/seed.js` (قيم `primary_color` و`secondary_color`) أو لاحقاً من صفحة إعدادات إضافية.

## خطوات التشغيل (سهلة جداً)

### 1) تنصيب المتطلبات
تحتاج جهاز فيه Node.js نسخة 18 أو أحدث (نزّله من nodejs.org إذا ما عندك).

```bash
cd branch-inspect
npm install
```

### 2) إعداد ملف البيئة
```bash
cp .env.example .env
```
افتح `.env` وغيّر `JWT_SECRET` لأي نص عشوائي طويل (مهم للأمان).

### 3) تجهيز قاعدة البيانات (أول مرة فقط)
```bash
npm run seed
```
هذا الأمر ينشئ قاعدة البيانات، ويعطيك مستخدم أدمن جاهز:
- **اسم المستخدم:** admin
- **كلمة المرور:** Admin@123

⚠️ غيّر كلمة المرور فوراً من صفحة "المستخدمون" بعد أول دخول.

### 4) التشغيل
للتجربة المحلية:
```bash
npm run dev
```
افتح المتصفح على: `http://localhost:3000`

للتشغيل الفعلي (إنتاج):
```bash
npm run build
npm run start
```

## أين أستضيفه؟
بما إنه ما عندك تفضيل، أسهل وأرخص طريقة:
- **Railway.app** أو **Render.com**: تربط المشروع مباشرة وتشغّله بدون إعداد سيرفر يدوياً، ويدعمون "Persistent Disk" لحفظ ملف قاعدة البيانات SQLite بشكل دائم (مهم جداً - تأكد تفعّل Persistent Volume على مسار `data/`).
- أو أي **VPS** بسيط (Hostinger, DigitalOcean...) وتشغّل عليه Node.js مباشرة مع PM2.
- إذا احتجت لاحقاً تتوسع لعدد مستخدمين كبير جداً ونظام أثقل، تقدر تستبدل SQLite بـ PostgreSQL بسهولة لأن كل الاستعلامات في `lib/db.js` منظمة ومركزية.

## ربط Google Sheets (تصدير إضافي)
كما اتفقنا، قاعدة البيانات الرئيسية SQLite للأمان والسرعة. لإضافة تصدير تلقائي لتقرير Google Sheets لاحقاً:
1. أنشئ مشروع في Google Cloud Console وفعّل Google Sheets API.
2. أنشئ Service Account واحصل على ملف JSON.
3. شارك الشيت مع بريد الـ Service Account.
4. أضف مكتبة `googleapis` وأنشئ API route جديد (`pages/api/export/sheets.js`) يقرأ من جدول `evaluations` ويكتب للشيت.

هذا الجزء يحتاج مفاتيح API خاصة بحسابك على Google، فتركته كخطوة تالية بعد ما تجهز حساب Google Cloud الخاص بالشركة — أقدر أكتبه لك جاهز فور ما توفر بيانات الحساب.

## ملاحظة مهمة عن ملف PDF
قالب PDF الحالي يعمل فوراً لكن بحروف إنجليزية/أرقام لضمان التوافق الكامل من أول تشغيل، لأن دعم اللغة العربية RTL في PDF يحتاج تضمين خط عربي (مثل Amiri أو Cairo) داخل المشروع. إذا تبيها عربي بالكامل داخل الـ PDF، قوللي وأجهزها لك بخط عربي مضمّن.

## إضافة قسم تقييم جديد لاحقاً
من داخل النظام: **أقسام التقييم → إضافة قسم جديد**. يظهر تلقائياً في كل نماذج التقييم القادمة بدون أي تعديل على الكود.

## إضافة فرع جديد
من داخل النظام: **الأفرع → إضافة فرع جديد**. لا حد لعدد الأفرع.

## هيكلة المشروع
```
branch-inspect/
  lib/          -> قاعدة البيانات، المصادقة، الإعدادات
  pages/api/    -> كل الـ API (الخلفية)
  pages/        -> كل الصفحات (الواجهة)
  components/   -> القائمة الجانبية والتخطيط العام
  styles/       -> التنسيقات
  data/app.db   -> ملف قاعدة البيانات (يُنشأ تلقائياً)
```
